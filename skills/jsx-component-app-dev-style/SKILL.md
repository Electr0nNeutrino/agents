# JSX/TSX 应用级组件开发范式

## 目标

为使用 JSX/TSX 的应用级项目（网站、应用程序等）提供标准化的组件开发流程，将组件实现拆分为**骨架与样式**、**操作与状态**两个阶段，配合通用规范，确保组件结构清晰、职责单一、状态管理合理。

## 适用场景

- 在 React / Solid / Preact 等 JSX/TSX 项目中开发新组件。
- 需要将设计稿转化为组件实现。
- 需要为组件引入或重构状态管理。
- 组件存在内部子组件需要合理组织文件结构。

## 阶段总览

各阶段可**独立执行**。用户可以只执行阶段一（如骨架已由他人完成则跳过），也可以直接从阶段二开始。无论执行哪个阶段，通用规范始终生效。

```
设计稿/需求
    │
    ▼
┌──────────────────────┐
│     通用规范(贯穿)     │
├──────────────────────┤
│  阶段一: 骨架与样式    │──▶ HTML + CSS + mock data
├──────────────────────┤
│  阶段二: 操作与状态    │──▶ 事件处理 + 状态管理
└──────────────────────┘
    │
    ▼
  完整组件

各阶段可单独执行，入口由用户指定。
```

## 通用规范

以下规范贯穿所有阶段，必须始终遵守。

### G1: 单文件组件

每个组件对应一个文件。

### G2: 组件目录定位

组件放置于 `components` 目录中。优先使用项目中已存在的 `components` 目录。如果不存在或存在多个名为 `components` / `component` 的目录，**必须**检查已实现的逻辑或询问用户意见后再做决定。

### G3: 类型同文件

组件 props 和组件实现中自定义的枚举等类型在组件文件内定义，**禁止**将其单独提取到外部文件。

### G4: 类型注释与文件说明

- 文件中所有类型**必须**使用 JSDoc 编写注释。
- 文件顶部使用注释说明该组件的功能。

### G5: 内部子组件拆分

实现组件时如果产生内部子组件（如为实现 A 而额外实现了 B），将其放置在两个单文件组件中，并按可重用性决定路径：

| B 的可重用性             | 文件结构                                     |
| ------------------------ | -------------------------------------------- |
| **低**（仅被 A 使用）    | `components/A/A.tsx` 与 `components/A/B.tsx` |
| **高**（被多个组件使用） | `components/A.tsx` 与 `components/B.tsx`     |

### G6: 实现方式决策

如果一种实现存在多种可能（包括但不限于状态管理方案、CSS 方案、路由方案等），根据以下优先级确定实现方式后再编写代码：

1. 已有说明文档（如 README 中指定的技术选型）
2. 先前的代码逻辑（已有的惯例与依赖）
3. 用户说明

例如：README 中已指定使用 jotai 作为全局状态管理，则不再考虑其他方案；项目中已使用 CSS Modules，则新组件也使用 CSS Modules。

## 阶段一: 骨架与样式

> 本阶段优先实现设计稿中的视觉要求，主要完成 HTML 和 CSS 的编写。状态相关逻辑优先级极低。

### S1: 仅使用基础状态

只使用 `useState` 或框架内同等作用的 hook（如 Solid 的 `createSignal`）。**禁止**使用自定义 hook 函数和第三方状态管理。

### S2: 提供 Mock Data

为便于与设计等人员沟通反馈，在组件文件内提供基础 mock data。命名格式为 `${组件名称}MockData`。

进入阶段二接入真实状态后，mock data **保留**在文件中，但添加注释标记以便区分：

```tsx
/**
 * Mock data - 仅用于开发调试与设计沟通，不参与生产逻辑。
 */
export const UserCardMockData = {
  name: "张三",
  avatar: "https://example.com/avatar.png",
  role: "管理员",
};
```

## 阶段二: 操作与状态

> 本阶段侧重完善状态管理与事件处理，样式变动占比很小。

### O1: React Compiler 检查（React 项目）

进入本阶段的**第一步**必须检查 React Compiler 的启用情况：

- **已启用** → 不需要考虑记忆化问题。
- **未启用** → 询问用户是否开启 React Compiler。如果不选择开启，后续流程中酌情进行记忆化处理（`useMemo`、`useCallback`、`React.memo`）。

### O2: 事件处理函数

根据逻辑复杂度决定是否提取辅助函数：

| 复杂度         | 处理方式                                               |
| -------------- | ------------------------------------------------------ |
| 简单（单行内） | 内联处理，如 `alert("todo!")` / `console.log("todo!")` |
| 复杂           | 提取为辅助函数，命名统一为 `handle${Xxx}`，使用小驼峰  |

### O3: 状态分级

分析状态是否需要全局化：

| 状态类型                                    | 处理方式                                     |
| ------------------------------------------- | -------------------------------------------- |
| 组件内部状态（如计数器的计数值）            | 保持在组件内部                               |
| 多个平行组件共用的状态（如主题、i18n 语言） | 全局化                                       |
| 与后端交互的状态                            | **必须**全局化                               |
| 子组件使用父组件状态                        | 优先多层 props 传递；超过 3 层时考虑 context |

### O4: 全局状态目录

全局状态放置在 `database` 或 `db` 路径下导出给组件使用。优先使用项目中已有的目录。如果不存在或存在多个同名目录，**必须**检查已实现的逻辑或询问用户意见后再做决定。

### O5: 全局状态内聚性

在保持内聚性和减少重复性的基础上将全局状态封装为单文件。例如多个组件都使用从服务端获取的用户信息（部分用名称、部分用邮箱），只需封装一个综合的用户信息全局状态，而非分别封装两次。

### O6: 全局状态接口

全局状态文件中需要提供：

- 该状态的 `set` 和 `get` 函数。
- 与服务端交互的逻辑也在该文件中编写。

## 流程总结

> 各阶段可独立执行。若用户指定了某一阶段，直接从该阶段开始，通用步骤（1-4）始终执行。

```
1. 定位 components 目录                        [G2]
2. 确认技术选型与实现方式                       [G6]
3. 创建组件文件，编写文件顶部说明               [G1, G4]
4. 定义 props 类型与 JSDoc 注释                 [G3, G4]
5. ── 阶段一: 骨架与样式 ──
   a. 实现 HTML 结构与 CSS 样式                 [S1]
   b. 提供 mock data                            [S2]
   c. 如产生子组件，按可重用性拆分文件           [G5]
6. ── 阶段二: 操作与状态 ──
   a. (React) 检查 React Compiler               [O1]
   b. 编写事件处理逻辑                          [O2]
   c. 分析并分级状态                            [O3]
   d. 全局状态放入 database/db 目录              [O4, O5, O6]
   e. 为 mock data 添加注释标记                  [S2]
7. 最终检查: 类型注释完整、文件结构合理
```
