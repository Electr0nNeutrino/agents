# TDD Workflow

## 目标

通过可重复执行的 TDD 流程，把需求拆解为一组小步可验证的行为，确保每次改动都由失败测试驱动，并在通过后进行安全重构。

## 适用场景

- 新增业务功能，需要明确行为边界。
- 修复线上缺陷，需要先复现再修复。
- 计划重构现有代码，需要测试作为安全网。
- 需要把模糊需求转为可执行验收标准。

## 输入信息

- 需求描述: 业务目标、输入输出、约束。
- 技术约束: 语言、框架、测试工具、性能/安全要求。
- 现有上下文: 相关模块、接口契约、历史缺陷。

## 输出结果

- 一组可读、可重复、可维护的自动化测试。
- 最小实现代码(仅满足当前测试)。
- 必要重构提交(不改变外部行为)。
- 完整验证记录: 全量测试通过、关键质量门禁通过。

## 核心流程 (Red -> Green -> Refactor)

1. 明确一个最小行为
2. 先写一个失败测试 (Red)
3. 运行测试并确认因预期原因失败
4. 写最小实现让测试通过 (Green)
5. 运行测试并确认通过
6. 重构测试与生产代码 (Refactor)
7. 运行全套相关测试，进入下一轮

## 详细步骤

### 1) 需求切片与验收定义

1. 将需求切分为最小行为单元(一次循环只做一件事)。
2. 为每个行为定义验收条件:
   - 正常路径
   - 边界条件
   - 异常路径
3. 写出测试命名草案，格式建议: `should_<result>_when_<condition>`。

完成检查:

- 每个行为可在 5-15 分钟内完成一轮 TDD。
- 测试名称能表达业务意图而非实现细节。

### 2) Red: 先写失败测试

1. 只为当前最小行为写测试，不提前实现后续逻辑。
2. 使用 Arrange-Act-Assert 结构组织测试。
3. 尽量只断言一个核心行为，避免多个失败原因混杂。
4. 对缺陷修复场景，先写“复现测试”。

完成检查:

- 测试可运行且稳定失败。
- 失败原因与预期一致(不是语法错误/环境错误)。

### 3) Green: 最小实现

1. 仅实现让当前测试通过所必需的最小逻辑。
2. 避免提前抽象、避免一次引入多个分支。
3. 必要时使用最简单硬编码通过当前测试，然后用下一轮测试逐步泛化。

完成检查:

- 当前新增测试通过。
- 未引入与当前行为无关的额外功能。

### 4) Refactor: 安全重构

1. 消除重复、提升可读性、优化命名与模块边界。
2. 若出现复杂条件分支，优先重构为更清晰的策略/映射。
3. 保持对外行为不变。

完成检查:

- 重构前后测试结果一致。
- 代码复杂度下降或可读性明显提升。

### 5) 回归与门禁

1. 运行受影响测试集(模块级)。
2. 运行全量关键测试(项目级)。
3. 执行静态检查(如 lint/type check)。
4. 检查覆盖率阈值(若项目定义了门禁)。

完成检查:

- 关键测试全部通过。
- 无新增 lint/type 错误。
- 覆盖率不低于既有基线(或达到目标阈值)。

## 决策点与分支逻辑

### 决策点 A: 测试写在什么层级?

- 如果行为是纯业务规则: 先单元测试。
- 如果行为依赖多模块协作: 增加集成测试。
- 如果行为是用户旅程或接口契约: 增加端到端/契约测试。

### 决策点 B: 先补测试还是先改代码?

- 新需求: 先写失败测试。
- 线上缺陷: 先写复现测试，再修复。
- 历史遗留且难以直接测试: 先做可测试性重构(小步且受现有测试保护)。

### 决策点 C: 一轮循环何时结束?

- 当前测试从失败变为通过。
- 已完成必要重构。
- 相关测试集通过。
  满足以上三项才进入下一轮。

## 质量标准

- 测试可读: 名称即需求、结构清晰、失败信息可定位。
- 测试可靠: 无随机失败、无环境耦合、可重复执行。
- 变更最小: 每轮只改必要代码，减少回归风险。
- 重构受保护: 任意重构必须以测试通过为前提。

## 常见反模式与纠正

- 先写实现后补测试 -> 回到 Red，从行为重新写失败测试。
- 一次写大量测试/大量实现 -> 缩小切片，保持小步快跑。
- 测试依赖实现细节 -> 改为面向可观察行为断言。
- 跳过重构 -> 在每轮 Green 后预留固定重构步骤。

## 执行模板

每一轮可按以下模板记录:

1. 行为: <一句话行为描述>
2. Red:
   - 新增测试: <测试名>
   - 失败原因: <预期失败>
3. Green:
   - 最小实现: <本轮改动点>
4. Refactor:
   - 重构项: <命名/抽取/结构调整>
5. 验证:
   - 相关测试: <通过/失败>
   - 全量关键测试: <通过/失败>

## 与用户协作时的提问顺序

当需求信息不足时，按顺序补齐:

1. 这次改动的业务结果是什么?
2. 成功/失败各自的可观察结果是什么?
3. 哪些边界条件必须覆盖?
4. 现有测试框架与运行命令是什么?
5. 是否有覆盖率、性能或安全门禁?

## 完成定义 (Definition of Done)

- 所有新增/修改行为均有对应自动化测试。
- 先失败后通过的证据完整。
- 代码经过至少一次重构清理。
- 相关测试、关键全量测试、静态检查全部通过。
- 变更说明清楚记录了行为、测试和风险。
