# Commit Helper

## 目标

根据[约定式提交 1.0.0](https://www.conventionalcommits.org/zh-hans/v1.0.0/) 规范，分析代码变更并生成结构化的提交说明（commit message），确保提交历史清晰、可自动化处理。描述（description）和正文（body）支持使用中文编写。

## 适用场景

- 提交任何代码变更，需要生成规范的提交说明。
- 不确定应使用哪种类型（type）或如何描述变更。
- 包含破坏性变更（BREAKING CHANGE）需要正确标记。
- 需要编写包含正文（body）和脚注（footer）的完整提交说明。

## 提交说明格式

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 脚注]
```

## 可用类型（Type）

| 类型       | 用途                                               | SemVer 关联 |
| ---------- | -------------------------------------------------- | ----------- |
| `feat`     | 新增功能                                           | MINOR       |
| `fix`      | 修复缺陷                                           | PATCH       |
| `docs`     | 修改文档，例如 README、API 文档等                  | -           |
| `style`    | 修改代码样式，例如调整缩进、空格、空行等           | -           |
| `refactor` | 重构代码，修改代码结构、变量名等但不修改功能逻辑   | -           |
| `perf`     | 优化性能，例如提升代码性能、减少内存占用等         | -           |
| `test`     | 修改测试用例，例如添加、删除、修改测试             | -           |
| `build`    | 修改项目构建系统，例如修改依赖库、升级 Node 版本等 | -           |
| `ci`       | 修改持续集成流程，例如修改 CI 工作流配置           | -           |
| `chore`    | 对非业务性代码进行修改，例如修改构建流程或工具配置 | -           |
| `revert`   | 还原（回滚）之前的提交                             | -           |

## 流程

### 1) 分析变更内容

1. 查看当前暂存区或工作区的变更 (`git diff --staged` 或 `git diff`)。
2. 识别变更涉及的模块/文件范围。
3. 判断变更的性质: 新功能、修复、重构、文档等。

### 2) 确定类型和范围

1. 根据变更性质从类型表中选择最匹配的类型。
2. 如果变更集中在某个模块，添加范围（scope），例如 `feat(auth):`。
3. 如果变更涉及破坏性改动，在类型/范围后加 `!`，例如 `feat(api)!:`。

决策要点:

- 一个提交只对应一种类型。如果变更涉及多种类型，建议拆分为多个提交。
- 范围应为一个名词，描述代码库的一个区域，例如 `parser`、`auth`、`api`。
- 类型使用小写英文。

### 3) 编写描述（Description）

1. 紧跟在 `类型[范围]:` 后的空格之后。
2. 可以使用中文或英文编写。
3. 中文描述：使用简洁的陈述句，例如"修复用户登录时 token 未刷新的问题"。
4. 英文描述：使用祈使语气（例如 "add validation" 而非 "added validation"），首字母不大写，末尾不加句号。
5. 保持简短，建议不超过 50 个字符。

### 4) 编写正文（Body，可选）

1. 与描述之间空一行。
2. 解释变更的动机和与之前行为的对比。
3. 自由格式，可使用中文编写，可包含多段。

### 5) 编写脚注（Footer，可选）

1. 与正文之间空一行。
2. 破坏性变更必须在脚注中标注: `BREAKING CHANGE: <说明>`（或在类型后使用 `!`）。
3. 关联 issue: `Refs: #123` 或 `Closes: #456`。
4. 脚注令牌（token）中用 `-` 替代空格（`BREAKING CHANGE` 例外）。

### 6) 输出与执行

1. 将生成的提交说明完整展示给用户确认。
2. 用户确认后，执行 `git commit -m "<message>"` 或对于多行消息使用 `git commit -F -`。

## 示例

单行简单提交（中文描述）:

```
docs: 修正 CHANGELOG 中的拼写错误
```

带范围（中文描述）:

```
feat(lang): 添加波兰语支持
```

破坏性变更:

```
feat(api)!: 产品发货时向客户发送邮件通知
```

完整多段提交（中文正文）:

```
fix: 修复请求竞态条件

引入请求 ID 并记录最新请求的引用，丢弃非最新请求的响应。

移除之前用于缓解竞态问题的超时机制，该机制现已不再需要。

Reviewed-by: Z
Refs: #123
```

还原提交:

```
revert: 还原面条事件相关变更

Refs: 676104e, a215868
```

## 质量检查

- [ ] 类型从允许列表中选择且为小写英文。
- [ ] 描述简洁明了（中文用陈述句，英文用祈使语气）。
- [ ] 破坏性变更通过 `!` 或 `BREAKING CHANGE:` 脚注正确标记。
- [ ] 正文与描述之间有空行分隔。
- [ ] 脚注与正文之间有空行分隔。
- [ ] 一个提交只做一件事。
