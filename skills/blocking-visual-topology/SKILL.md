# 文件级可视化拓扑（Visual Topology）

> **核心公理**：每个关键文件的架构必须在打开文件的瞬间可见——不读代码即知全貌。
> 配置/代码文件：图在顶部注释块。文档：图在架构段内联。

---

## 一、放置规则

| 文件类型 | 位置 | 格式 |
|----------|------|------|
| **配置文件**（`config.yaml` 等） | 文件顶部，第一行开始 | 语言注释块（`#` / `//` / `/* */`） |
| **代码入口**（`main.go` / `cmd/main.go` 等） | `package` 声明之后、`import` 之前 | 块注释（`/* */` / `//`） |
| **文档**（`README.md` 等） | 架构段内联（如 `## Architecture`） | Markdown 代码块（` ```  ``` `） |

---

## 二、必须绘制的触发条件

以下场景**必须**绘制拓扑图：

| # | 触发条件 | 说明 |
|---|----------|------|
| V1 | **新建配置文件** | 任何含数据流/组件编排的配置文件 |
| V2 | **新建代码入口** | `main` 函数所在文件，展示启动→组件→数据流 |
| V3 | **组件拓扑变更** | 新增/删除/重排组件的调用顺序、输入、输出 |
| V4 | **README 架构段** | 编写或重构 README 中的架构/数据流章节 |
| V5 | **多组件交互** | 文件涉及 3+ 组件协作时，图比文字更高效 |

## 三、绘制规范

### 3.1 字符集

| 用途 | 字符 | 示例 |
|------|------|------|
| 水平线 | `─` (U+2500) 或 `-` | `───▶` |
| 垂直线 | `│` (U+2502) 或 `\|` | `│` |
| 箭头 | `▶` `▼` `→` `↓` | `──▶ Pipeline` |
| 方框 | `╔ ╗ ╚ ╝ ║` 或 `+--+` | 分组边界 |
| 分支 | `├ ┤ ┬ ┴ ┼` | 分叉/合流 |
| 标注 | `()` `[]` | `[plugins]` `(key=id)` |

### 3.2 布局原则

| # | 规则 |
|---|------|
| L1 | **从左到右**为主流向（入口在左，出口在右） |
| L2 | **并行分支垂直排列**，用方框框住同层管线组 |
| L3 | **合流点明确标注**合并策略（`union` / `join` + key） |
| L4 | **节点标注格式**：`名称 [插件列表] → 出口` |
| L5 | **对齐**：同层节点的箭头、方括号、出口名纵向对齐 |
| L6 | **最大宽度 80 字符**，超长时折行或缩写 |

### 3.3 标注约定

| 元素 | 标注方式 | 示例 |
|------|----------|------|
| 组件调用 | `名称 [组件名...]` | `fingerprint [simhash-fp]` |
| 零组件调用 | `名称 []` | `base []` |
| 出口 | `→ 出口名` | `→ kafka_base` |
| 合并器 | `策略(参数)` | `union(key=tweet_id)` |
| 引用 | `@pipeline/id` / `@merger/id` | `@merger/db_aggregate` |

## 五、变更同步规则

| # | 规则 | 说明 |
|---|------|------|
| S1 | **拓扑变更 = 图必更新** | 新增/删除/重排管线、出口、合并器后，顶部图必须同步修改 |
| S2 | **图-配置一致性** | 图中每个节点必须在配置正文中有对应声明；反之亦然 |
| S3 | **跨文件同步** | 同一拓扑在 config.yaml 和 README.md 中都有图时，两处必须一致 |
| S4 | **验证方法** | 自查：数图中节点数 = 数配置中组件/出口/合并器数，不一致即违规 |
